# 梯度累计
Gradient Accumulation：把多个小 batch 的梯度累加起来，再统一更新一次模型参数
梯度累计的意义：
**显存换时间
支持超大模型训练
稳定训练**
大 batch梯度更稳定方差更小，梯度累计可间接提高训练稳定性。
配合分布式训练
在 DDP 中，如果每次 backward 都 allreduce会通信爆炸，很多框架会accumulation 期间不通信，最后一次再同步梯度，这极大减少通信开销。

# 激活值（Activation）与激活重计算（Activation Checkpointing  Recomputation）

激活值：上一层输出与当前层计算出的**当前层输出**就叫**Activation（激活值）**，也叫activation tensor、hidden states，本质是前向传播过程中产生的中间结果，不是参数不是梯度，而是**为了反向传播必须临时保存的数据**
## Transformer 中的 Activation

├── QKV Projection 
├── Attention 
├── Softmax

这中间的 tensor都属于 activation，例如：

Activation 才是最大显存来源，尤其：长序列、大 batch、深层网络的情况下。

## Activation Recomputation
核心思想：forward 时，不保存全部 activation，只保存少量 checkpoint（通常是每2~4层保存一个checkpoint）。backward 时，缺失的 activation重新 forward 计算一遍。本质：**用额外计算换显存**
PyTorch checkpoint 原理
```
from torch.utils.checkpoint import checkpoint  
  
out = checkpoint(block, x)
```
forward 不保存中间activation，只保存：输入和RNG state（因为dropout有随机性）；backward 重新 forward，再计算梯度。
## 真正工业界使用的是“Segment Checkpoint”
Megatron / DeepSpeed / PyTorch真正使用的方法。
backward ≈ 2× forward FLOPs。因为很多算子backward实际包含两次 GEMM，而 forward只有一次。
例如：
Linear：Y=XW，一次 matmul；
而backward：weight grad![[Pasted-image-20260521210327.png]]input gra![[Pasted-image-20260521210347.png]]
**checkpoint 通常只 checkpoint Transformer Block**，因为每个 block结构一样，Megatron本质就是checkpoint(block)，且通常只 checkpoint attention，**重算 attention 比保存 attention 更划算**
GPU的算力增长远超HBM带宽增长，因此工业界倾向**少存，多算**，重新计算可能比HBM IO还便宜，这是FlashAttention、PagedAttention、Selective Recomputation等所有现代kernel优化的共同哲学。

# 优化器（Optimizer）

深度学习训练中的参数更新策略系统，决定：
- 参数怎么更新
- 更新方向是否稳定
- 学习率如何缩放
- 梯度是否可信
- 混精下数值是否崩坏
- 大batch是否还能收敛
## 优化器状态
Optimizer 不只是一个函数，它本身是一个“状态机”，optimizer 内部会保存大量历史统计信息，这些就叫**optimizer states（优化器状态）**
Adam不是只看当前梯度，而是同时看：
当前梯度 g、历史梯度均值（一阶动量）m、历史梯度平方均值（二阶动量）v
**m（momentum）本质是梯度方向惯性**：类似物理中的速度，如果：最近很多步都朝同一方向，Adam则会加速前进。
**v（variance）本质是梯度波动程度**：如果某维梯度变化剧烈说明不稳定，则Adam自动降低该维更新步长。
**Adam 本质其实是带自适应学习率的动量系统**![[Pasted-image-20260521211851.png]]
## optimizer states 通常用 fp32

这是混精核心，优化器里的统计量对数值精度极其敏感，更新被量化误差吞掉。
## 混合精度训练 iteration 中 Optimizer 怎么工作
### Step1 Forward
使用bf16 weight计算activation/loss。
### Step2 Backward
backward得到gradient。
### Step3 Gradient Scaling（混精常见）
梯度缩放防止underflow（下溢）。
### Step4 Optimizer Step
optimizer 在 fp32 master weight 上更新，Adam读取当前grad、m、v、master weight，然后依次更新m、v、计算 adaptive update、更新 master weight
### Step5 Weight Cast
更新后的 fp32 weight cast 成 bf16，供下一轮 forward。
### Step6 Zero Grad
optimizer.zero_grad()清空梯度缓存。
## 从 AI Infra 角度重新理解 Optimizer
现代大模型训练中，真正的显存大户往往是：activation、optimizer state，而不是参数本身。
因此大量 infra 技术，本质上都在解决 optimizer memory 问题：
- ZeRO Stage1：切 optimizer state
- CPU Offload：optimizer 放 CPU
- NVMe Offload：optimizer 放磁盘
- 8bit optimizer：压缩 m/v
- Paged optimizer：分页管理 optimizer

# 集合通信及其通信原语
## Broadcast
数据的1对多的同步![[Pasted-image-20260521195943.png]]
应用场景有：
1）数据并行的参数初始化，确保每张卡上的初始参数是一致的；
2）allReduce里的 broadcast + reduce组合里的broadcast操作；
3）分布式训练parameter server 参数服务器结构里的 master节点 broadcast 数据到worker节点，再从worker节点reduce数据回master节点里的broadcast操作；
## Scatter
一对多原语，将数据的进行切片再分发给集群内所有的节点，他的反向操作对应Gather![[Pasted-image-20260521200048.png]]
应用场景有：
1）ReduceScatter组合里的 Scatter操作；
2）模型并行里初始化时将模型scatter到不同的XPU上；
## Gather
多对1通信原语，多个数据发送者，一个数据接收者，可以在集群内把多个节点的数据收集到一个节点上，其反向操作对应Scatter![[Pasted-image-20260521200059.png]]
应用场景有：
1）ReduceScatter组合里的 Scatter操作；
## AllGather

多对多通信原语，具有多个数据发送者，多个数据接收者，可以在集群内把多个节点的数据收集到一个主节点上（Gather），再把这个收集到的数据分发到其他节点上（broadcast），即收集集群内所有的数据到所有的节点上（Gather + Broadcast）。反向操作对应ReduceScatter![[Pasted-image-20260521200108.png]]
应用场景有：
1） AllGather可应用于模型并行；
2）模型并行里前向计算里的参数全同步，需要用AllGather把模型并行里将切分到不同的XPU上的参数全同步到一张XPU上才能进行前向计算。
## Reduce

多对1通信原语，具有多个数据发送者，一个数据接收者，可以在集群内把多个节点的数据**规约运算**到一个主节点上，常用的规约操作符有：累加和SUM、累乘积PROD、最大值MAX、最小值MIN......![[Pasted-image-20260521200114.png]]
应用场景有：
1）AllReduce里的 broadcast + reduce组合里的reduce操作；
2）ReduceScatter组合里的 reduce操作；
3）分布式训练parameter server 参数服务器结构里的 master节点 broadcast 数据到worker节点，再从worker节点reduce数据回master节点里的reduce操作；
## ReduceScatter

多对多通信原语，具有多个数据发送者，多个数据接收者，其在集群内的所有节点上都**按维度执行相同的Reduce规约运算**，再将结果发散到集群内所有的节点上，Reduce-scatter等价于**节点个数次的reduce规约运算操作，再后面执行节点个数的scatter次操作**，其反向操作是AllGather。![[Pasted-image-20260521200121.png]]
应用场景有：
1）ReduceScatter即可应用于数据并行也可应用于模型并行；
2）数据并行allReduce里的 ReduceScatter+ Allgather组合里的ReduceScatter操作；
3）模型并行里在前向allgather后的反向计算里的ReduceScatter；
## AllReduce

多对多通信原语，具有多个数据发送者，多个数据接收者，其在集群内的所有节点上都执行相同的Reduce操作，可以将集群内所有节点的数据**规约运算**得到的结果发送到所有的节点上。AllReduce操作可通过在主节点上执行Reduce + Broadcast或ReduceScatter + AllGather实现![[Pasted-image-20260521200137.png]]
应用场景有：
1） AllReduce应用于数据并行；
2）数据并行各种通信拓扑结构比如Ring allReduce、Tree allReduce里的 allReduce操作；

## All-To-All

All-To-All操作每一个节点的数据会scatter到集群内所有节点上，同时每一个节点也会Gather集群内所有节点的数据。All-To-All是对AllGather的扩展，区别是AllGather操作中，不同节点向某一节点收集到的数据是相同的，而在All-To-All中，不同的节点向某一节点收集到的数据是不同的
**可以理解成是一种“转置式”地通信方法**![[Pasted-image-20260521200244.png]]

应用场景有：

2）模型并行里的矩阵转置；
3）数据并行到模型并行的矩阵转置；

# MFU（Model FLOPs Utilization，模型算力利用率）
MFU = 你实际跑出的 FLOPS ÷ 显卡理论峰值 FLOPS，Roofline 给了MFU 的理论上限 ，直观体现了**显卡的计算核心到底有多忙**$$MFU = 模型实际执行的 FLOPs / GPU 理论峰值 FLOPs$$
提高MFU的直接目标：**最直接地目标就是提升GPU使用率，充分发挥GPU的计算潜力，以加快模型训练**![[Pasted-image-20260603220943.png]]
## MFU 计算原理
### 手动推导FLOPs原则：
- 手动推导模型的FLOPs时只推导前向传播，大部分情况默认模型后向传播的计算量是前向传播的**2倍，** 总共FLOPs是前向的**3倍**。
- 由于LLM模型参数过大，占用显存过多，有时候为了降低显存在训练采用将中间参数保留在内存里——激活重计算。因此推导LLM训练时FLOPs如果考虑到中间参数的**激活重计算**的过程，需要计算整体FLOPs需要再加一份前向计算量，即1(前向） + 2(反向）+ 1(激活重计算）= **4 倍** 计算量。
- 手动推导模型的FLOPs时，优先推导整个过程计算量占大头部分，通常忽略激活函数、layer normalize，softmax等等部分计算量。
### 矩阵计算方法 
对于一般化的 $[m,n]×[n,p]$ 矩阵，GPU 两层循环分别计算行列，因此 $m×p$ 次行列运算，共 2mnp 次运算。![[Pasted-image-20260603215009.png]]

一次训练迭代包含了前向传递和后向传递，**后向传递的计算量是前向传递的2倍**。一次训练迭代中，对于每个token，每个模型参数，需要进行 $2*3=6$ 次浮点数运算。

## Roofline Model曲线图![[Pasted-image-20260603162723.png]]
图中的 $P$ 代表Attainable Performanc，代表实际性能，单位 $FLOPS/s$。； $I$ 代表Operational Intensity，单位 $FLOPS/Byte$，意思是从显存读取1 Byte数据，能产生多少次浮点计算。
- 当 $I < I_{max}$ 的时候，处于Memory Bound区域；此时MFU天然就低，上不去
- 当 $I > I_{max}$ 的时候，处于Compute Bound区域；只要 $I$ 足够大， $P$ 肯定是可以打满的

还是举个例子来看：
RTX3090 算力$π = 71.2TFLOPs$、带宽$β = 936GB/s$，$Imax​=π​/β=71.2×10^12/936×10^9=76$
如果：I>76，说明计算能力先耗尽，Compute Bound，GPU忙着算。如果I<76，说明带宽先耗尽，Memory Bound，GPU在等数据。
Hidden_Dim = 4096、假设Batch_Size * SeqLen = 256（Batch_Size = 2 ，Seq_Len = 128）
### Prefill阶段的 QKV Gemm
Prefill（上下文编码阶段），模型要一次性处理所有输入 Token 同时进入 Transformer，一次把整个 Prompt 编码完成。Decode（逐Token生成阶段），模型只生成1 个 token，然后再生成下一个 token，Decode阶段每次S=1。实际上通常Prefill只占20%，Decode占80%。原因是GPU特别喜欢大矩阵乘法，而特别讨厌小矩阵乘法。
QKV Projection：$$[B×S,H]×[H,H]$$
- 此时 $I = 2*128*4096*4096 / (128*4096+4096*4096+128*4096) ≈ 256$
- 可以看到 $I = 256 > I_{max}$ 的，所以Prefill阶段就是个compute bound，MFU高
- Prefill：很多token一起算，矩阵大，Arithmetic Intensity高，Tensor Core利用率高。
### Decode阶段的QKV Gemm
- Decode一次只生成一个 token，$B×S=1$
- 此时 $I = 2*1*4096*4096 / (1*4096+4096*4096+1*4096) ≈ 2$
- 可以看到 $I = 2 < I_{max}$ 的，所以Decode阶段就是个memory bound，MFU低
- GPU的大部分时间都在等待从HBM读4096×4096权重
- Decode：每次只算1个token，矩阵退化，Arithmetic Intensity低，GPU算力浪费。
这也是**vLLM、PagedAttention、Continuous Batching**都在拼命解决的问题，本质就是
**让 Decode 看起来更像 Prefill**。
## MFU 的计算方式
这里要区分：理论模型 FLOPs和实际训练时间，对于 Transformer，业界常用近似公式：$$FLOPs = 6 × N × T$$
其中：
- N：模型参数量，计算中前向约2N，反向约4N
- T：训练 token 数
这个公式来自 Transformer Forward+Backward 的近似计算量。
例如：7B 模型一次训练处理1M tokens，则：$$6×7B×1M=4.2×10^{16} FLOPs$$
如果这一步训练耗时10秒，则实际 FLOPs/s：$$4.2×10^{15} FLOPs/s$$即可得到 MFU。
## MFU 为什么通常只有 30%-50%，浪费来源在哪儿
GPU 大量时间并不在算。主要浪费来源：
1. 通信（GPU 在等待网络，此时MFU下降）
	- DDP 的 AllReduce。
	- Tensor Parallel 的 AllGather。
	- Pipeline Parallel 的 Send/Recv。
2. Activation IO（写/读显存，显存带宽受限）
	- 训练时大量 activation需要写/读显存
3. Kernel Launch 开销（Kernel 之间有调度开销）
	- Transformer 实际不是一个大矩阵乘法，而是几十甚至上百个 CUDA Kernel，Kernel 之间有调度开销
4. 数据加载
	- CPU读取数据→Tokenize→拷贝到 GPU→训练
5. Activation Checkpoint
	- Checkpoint需要额外 Recompute
## HFU 和 MFU 的区别
HFU：Hardware FLOPs Utilization $$HFU==\frac{实际硬件执行FLOPs}{理论峰值FLOPs}$$
MFU：$$MFU = \frac{有效模型 FLOPs}{理论峰值 FLOPs}$$
例如：正常训练总300 FLOPs，Checkpoint重算100 FLOPs，总400 FLOPs
HFU会变高，因为GPU 更忙了。但MFU 不变，因为真正模型需要的计算还是300 FLOPs。
## MFU 和吞吐量的关系
MFU 本质上和Tokens/s强相关。对于固定模型：$$MFU∝Tokens/s$$如果MFU 提高，训练速度就提高，训练成本就下降。
## 提升MFU的方法
### Prefill
$I$ 远大于拐点（算力受限）→ MFU 提升核心是**喂满 GPU 计算单元，减少算力闲置**
- 方法1：增大Batch Size，最直接，GPU并行计算的token越多，MFU越接近理论上限
- 方法2：多卡DP，但是单卡的Batch Size还是足够大，避免算力闲置
- 方法3：算子融合，减少无意义的显存读写，间接提升算力利用率（如果通信时间可以被计算时间完全掩盖的话能流水起来算子融合的意义就没那么大了）

```text
有点反直觉是吧，一般来说为了保证计算单元满载，我们需要数据量够大，那为什么还要拆DP呢？
-> GPU 计算单元的「最优并行粒度」，单卡任务量不是越大越好，超过阈值会因资源竞争导致利用率下降
-> 拆 DP 是让每张卡的任务量落在「最优区间」
-> 反过来想：单卡任务量太大有什么不好的地方？
  -> 资源竞争：GPU SM内的线程块抢寄存器 / 共享内存，部分任务要排队
  -> 调度延迟：GPU 内核要花更多时间分配任务，计算单元有 10%~20% 时间闲置
```
### Decode
$I$ 远小于拐点（带宽受限）→ MFU 提升核心是**拉高 $I$ 值，让实际 FLOPS 逼近带宽上限**
- 方法1：增大Batch Size；Decode 阶段 I≈2×batch，batch 越大，I 越高，MFU 越接近理论上限
- 方案2：用更低精度的量化方案 -> 减少访存，变向增加可执行的Batch数量，提升I
- 方案3：通过片上share memory来缓存权重，减少重复访存的开销（Lx-Cache思路）

### 训练优化
- 方法1：Communication Overlap
	- 例如使用：ZeRO、FSDP、通信重叠（Overlap），减少 GPU 等待时间。
- 方法2：增大 GEMM 规模
	- GPU 最擅长大矩阵乘法，增大Batch Size、Sequence Length，都能提高 Tensor Core 利用率。通常Global Batch 越大，MFU 越高。
	- Batchb不是越大越好，GPU存在最佳工作区间，比如Batch=64已经让 Tensor Core 满载，继续增大Batch=128、256，MFU提升很小，反而activation暴涨。batch需要找到Tensor Core饱和点
- 方法3：FlashAttention
	- FlashAttention 的本质：不是减少 FLOPs，而是减少显存 IO（HBM Traffic），提高计算密度。因此MFU 通常明显提升。
- 方法4：Kernel Fusion——减少 Kernel 数量
	- 把：LayerNorm+Dropout+Residual Add融合成一个 CUDA Kernel。减少Kernel Launch。Megatron-LM 大量使用这种优化。
	- Transformer里 LayerNorm、Dropout、Bias Add、Residual Add 本来4个 CUDA Kernel，意味着：Kernel Launch→读显存→写显存→Kernel Launch→读显存→写显存，数学计算量很少、IO很多。
	- Megatron、Apex、TransformerEngine大量做Fused LayerNormFused BiasAddFused Softmax，本质是减少HBM IO
- 方法5：Activation Checkpoint 的取舍
	- Checkpoint 的目标从来不是提升 MFU，而是省显存，然后允许更大的Batch更长的Seq，间接提高 MFU、提高Arithmetic Intensity。
- 方法6：更好的并行策略
	- 训练系统设计的核心是**让并行扩展后 MFU 不掉太多**
	- Data Parallel：MFU 通常最高。
	- Tensor Parallel：通信增加，MFU下降。
	- Pipeline Parallel：出现 bubble，MFU下降。
	并行策略设计本质上也是MFU 优化问题。

# 吞吐 Throughput
网络模型在单位时间内可以处理的最大输入的训练样本数据。
与涉及单个样本数据处理的延迟Latency 不同，为了实现最大吞吐量Throughput，希望在集群训练的过程中并行处理尽可能多的样本数据。有效的并行性显然依赖于数据、模型和设备规模。
## 吞吐量测量与计算
可以执行以下两个步骤：
1. 估计允许最大并行度的最佳训练样本数据批量大小，即Batch Size
	要找到最佳批量大小，一个好的经验法则是达到Ascend昇腾处理器+对给定数据类型的内存限制，即BatchSize塞满内存。这个大小当然取决于硬件类型和网络的大小。
2. 在AI训练集群中给定这个最佳批量大小，测量网络在1秒钟内可以处理的训练样本数据。

这确定了AI加速卡上可以处理的最大批量大小，用于训练大模型及其处理的输入训练样本数据。在找到最佳批量大小后，可以计算实际吞吐量，然后使用以下公式：$${sample}_{throughput} = BS*N/{step}_{time}$$
其中，$BS$ 为Batch Size，$N$ 为集群中AI加速卡的数量，step time为在分布式集群中，执行完一个Batch Size的时间（单位为s），包含前向传播、反向传播、梯度同步（Communication）以及数据加载（I/O）的总耗时。大模型训练的吞吐率的单位在CV和部分NLP任务中为Samples/s，在固定 Shape 的 NLP任务有seq len，因此单位为tokens/s，计算公式为：$${tokens}_{throughput} = {sample}_{throughput}*{seq}_{len}$$
假设 GLM 10B 网络模型的吞吐为25 Samples/s，max seq_len为1024，那么按照tokens来计算吞吐率为 $25*1024=25600tokens/s$ 。也就是每秒能处理2万多个tokens啦。





# Flash Attention
FlashAttention是一种**精确的优化算法**， Transformer 的瓶颈根本不是 FLOPs，而是 Memory IO。FlashAttention 之后，大家开始意识到，怎么减少HBM访问可能比减少计算更重要。

标准的缩放点积注意力（Scaled Dot-Product Attention）公式如下： $$\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V$$
标准Transformer的计算一共包含八次HBM的矩阵读写操作。这八次读写操作分别为：
- 对 $Q,K$ 的读取共两次，对 $S = QK^T$ 的写入一次，读写总共三次；
- 第二行对 $S$ 读取一次，对 $P = softmax(S)$ 写入一次，读写总共两次；
- 第一行对 $P,V$ 的读取共两次，对 $O = PV$ 的写入一次，读写总共三次。
为了减少对HBM的读写，FlashAttention将参与计算的矩阵进行**分块**送进SRAM，来提高整体读写速度（减少了HBM读写）

## Flash Attention v1
对于矩阵乘法而言，可以直接通过分块来达到分块计算的目的。但Self-Attention中有 $softmax$ 计算，而 $softmax$ 的分母包含与所有元素相关的求和项，所以对Self-Attention进行分块计算的真正难点在于对 $softmax$ 的分块计算。
